## 1. 기획

### ■ 설정한 사용자는 누구이며, 해결하려는 구체적인 문제점/불편함은 무엇인가요?

**메인 사용자:** 국비지원/부트캠프 코딩 강사
**서브 사용자:** 수강생 (비전공자 포함)

**해결하려는 구체적 문제:**

현재 모든 코딩 교육은 "코드를 짜는 법"만 가르치고, "코드를 읽고 고치는 법(디버깅)"은 체계적으로 가르치지 않는다. 그러나 실무 개발자의 하루 60~70%는 기존 코드를 읽고, 버그를 찾고, 수정하는 데 소비된다(대규모 코드베이스에서는 80~90%).

이로 인해 세 가지 구체적 문제가 발생한다:

1. **수강생의 문제:** 부트캠프에서 코드 짜는 건 배웠지만, 실무에서 에러를 만나면 대처를 못 한다. 실제로 "브레이크포인트도 못 잡는 신입 개발자"라는 현장 피드백이 반복된다. 에러가 나면 ChatGPT에 코드를 통째로 붙여넣어 답을 받을 뿐, 스스로 원인을 찾아 해결하는 사고력이 훈련되지 않는다.

2. **강사의 문제:** 디버깅은 사고 과정이 핵심인데, 강사 1명이 30명 학생의 디버깅 과정을 일일이 관찰하고 피드백하는 것은 물리적으로 불가능하다. 또한 디버깅 실습 문제를 수동으로 만드는 것은 시간이 오래 걸린다.

3. **AI 시대의 구조적 문제:** GitHub Copilot, Cursor 등 AI 코딩 도구가 코드 작성을 대체하면서, 개발자에게 남는 핵심 역량은 "AI가 생성한 코드를 검증하고, 버그를 찾아 수정하는 능력"이다. 그러나 현재 교육 체계는 이 변화를 전혀 반영하지 못하고 있다.

### ■ 문제를 해결하기 위한 솔루션의 핵심 기능은 무엇인가요?

BugHunter는 AI가 의도적으로 버그가 포함된 코드를 생성하고, 학생이 이를 찾아 고치는 과정에서 디버깅 사고력을 훈련하는 교육 플랫폼이다.

**핵심 기능 3가지:**

**1) AI 버그 코드 자동 생성**
강사가 수업 주제(예: "반복문")만 입력하면, AI가 해당 주제와 관련된 버그가 포함된 Python 코드를 3단계 난이도(초급/중급/고급)로 자동 생성한다. 초급은 문법/인덱스 오류, 중급은 로직 오류(무한루프, off-by-one), 고급은 엣지케이스 미처리 등 실무에서 실제로 발생하는 버그 유형을 다룬다. 각 문제에는 정확히 버그 1개만 포함되며, 테스트케이스가 함께 생성되어 정답 검증이 가능하다.

**2) 단계별 힌트 시스템 (답을 주지 않는 AI)**
학생이 막혔을 때, AI는 정답을 직접 알려주지 않고 3단계의 점진적 힌트를 통해 스스로 사고하도록 유도한다.
- 1단계(방향 제시): "반복문 부분을 확인해보세요" — 문제 영역만 안내
- 2단계(범위 좁히기): "3번째 줄의 range 함수 범위를 확인해보세요" — 구체적 위치 안내, 수정 방법은 미제공
- 3단계(개념 설명): "이것은 off-by-one 에러입니다. range()의 끝 값은 포함되지 않습니다" — 원인 설명 + 수정 방향 제시

이 구조는 ChatGPT(바로 정답 제공)와 근본적으로 다르며, "스캐폴딩(scaffolding)" 교육 이론에 기반한 설계이다.

**3) 강사 대시보드 + 학생 성장 리포트**
강사는 대시보드에서 반 전체의 디버깅 현황을 한눈에 파악한다. "반복문 버그는 80%가 잡지만, 조건문 엣지케이스는 30%만 잡는다" 같은 인사이트를 통해 다음 수업 계획에 반영할 수 있다. 학생은 자신의 디버깅 성장 기록(레벨 변화, 약점 영역)을 확인하며 동기를 유지한다.

### ■ 이 솔루션이 도입되었을 때 기대되는 개선점이 무엇인가요?

1. **강사 업무 효율화:** 디버깅 실습 문제를 수동으로 제작하는 시간이 "주제 입력 10초"로 단축된다. AI가 30명 학생에게 개별 힌트를 제공하므로, 강사는 AI가 커버하지 못하는 소수의 학생에게만 집중할 수 있다(AI 25명 + 강사 5명 구조).

2. **수강생 디버깅 역량 향상:** 기존에는 "에러 → ChatGPT 복붙 → 정답 받기"로 끝나던 학습이, "에러 → 힌트 기반 자기 사고 → 직접 해결"로 전환된다. CMU 연구(HypoCompass)에서 이 방식이 학생 성적을 12% 향상시킨 것으로 입증되었다.

3. **교육기관 취업률 개선:** 수료생이 실무에서 가장 많이 부딪히는 "디버깅"을 체계적으로 훈련함으로써, 채용 기업의 만족도가 올라가고 교육기관의 취업률 KPI가 개선된다.

4. **AI 시대 교육 패러다임 전환:** "코드를 짜는 교육"에서 "코드를 검증하고 고치는 교육"으로의 전환을 선도한다.

## 2. AI 활용 전략

### ■ 이 프로젝트에서 사용할 AI 도구(Claude Code, Cursor, Codex 등)와 모델은 무엇이며, 선택한 이유는 무엇인가요?

| AI 도구/모델 | 용도 | 선택 이유 |
|-------------|------|-----------|
| **Google Gemini 2.0 Flash** | 버그 코드 생성, 힌트 생성, 코드 분석 | 빠른 응답 속도, 우수한 코드 이해력, 무료 API 제공으로 MVP 단계에서 비용 부담 없음. 코드 생성 및 분석 품질이 교육용으로 충분 |
| **Claude Code** | 프로젝트 개발 전반 | 풀스택 웹 개발(Next.js + Supabase)의 코드 작성, 디버깅, 프롬프트 엔지니어링에 활용. 대규모 컨텍스트 처리 능력이 뛰어나 프로젝트 전체를 이해하며 개발 가능 |

### ■ 각 AI 도구별 활용 전략을 작성해주세요.

#### Gemini 2.0 Flash — 서비스 핵심 AI 엔진

**페르소나:** "프로그래밍 교육 전문가이자 Python 코드 버그 생성기"

**핵심 전략:**
BugHunter에서 Gemini는 3가지 역할을 수행하며, 각각 전문화된 프롬프트로 동작한다:

**1) 버그 코드 생성 에이전트**
- 입력: 수업 주제 (예: "반복문")
- 출력: 3단계 난이도의 버그 챌린지 (JSON 배열)
- 프롬프트 설계: 각 난이도별 버그 유형을 명시적으로 제약(beginner=문법 오류, intermediate=로직 오류, advanced=엣지케이스). "버그 정확히 1개만 포함" 규칙을 강제하여 학습 목적에 맞는 문제 품질을 보장한다.
- 응답 형식: 엄격한 JSON 스키마를 지정하여 파싱 오류를 방지. title, difficulty, buggy_code, correct_code, bug_description, instructions, test_cases 필드를 포함한다.

**2) 힌트 생성 에이전트**
- 입력: 원본 버그 코드 + 정답 코드 + 학생 현재 코드 + 힌트 레벨(1/2/3)
- 출력: 해당 레벨에 맞는 교육적 힌트 (자연어 텍스트)
- 프롬프트 설계: 3단계 힌트 각각에 대해 "무엇을 알려주고, 무엇을 알려주지 않는지"를 명확히 제약한다.
  - Level 1: 영역만 안내, 줄 번호/변수명 금지
  - Level 2: 구체적 줄 안내, 수정 방법 금지
  - Level 3: 원인 설명 + 수정 방향, 전체 정답 코드 금지
- 제약 사항: "절대 정답 코드를 보여주지 마"를 모든 레벨에서 강제. 이것이 ChatGPT와의 핵심 차별점이다.

**3) 솔루션 검증 에이전트**
- 입력: 학생이 수정한 코드 + 원본 정답 코드
- 출력: 코드 정확성 판단 + 피드백
- Pyodide(브라우저 내 Python 실행)로 테스트케이스를 실행한 후, AI가 추가 피드백을 제공하는 이중 검증 구조.

**데이터 흐름:**
```
강사: 주제 입력 → [버그 생성 에이전트] → 3단계 버그 챌린지 생성 → DB 저장
학생: 문제 풀기 → 막힘 → [힌트 에이전트] → 레벨별 힌트 제공
학생: 코드 수정 → 제출 → [Pyodide 실행 + 검증 에이전트] → 결과 + 피드백
결과 → DB 저장 → 강사 대시보드 + 학생 성장 리포트
```

#### Claude Code — 개발 도구

**핵심 전략:**
- 프로젝트 전체 코드베이스를 컨텍스트로 로드하여, 일관성 있는 풀스택 개발 수행
- Next.js App Router, Supabase 연동, API Route 설계 등 프로젝트 아키텍처 전반을 관리
- 프롬프트 엔지니어링 테스트: Gemini에 전달할 프롬프트의 품질을 반복 테스트하며 최적화
- 에러 디버깅: 개발 중 발생하는 빌드 에러, 타입 에러를 빠르게 진단 및 수정

### ■ 토큰 낭비를 최소화하고 유지보수성 및 재현성을 높이기 위한 전략이 있다면 작성해주세요.

**1) 프롬프트 모듈화**
버그 생성 프롬프트와 힌트 생성 프롬프트를 별도 모듈(`/lib/prompts/generate-bugs.ts`, `/lib/prompts/generate-hint.ts`)로 분리하여 관리한다. 각 프롬프트는 함수 형태로 작성되어, 입력 파라미터만 바꾸면 동일한 품질의 결과를 재현할 수 있다.

**2) 구조화된 JSON 응답 강제**
AI 응답을 자유 텍스트가 아닌 엄격한 JSON 스키마로 강제하여, 파싱 오류를 방지하고 후처리 비용을 절감한다. "JSON 배열만 응답해. 다른 텍스트 없이."를 프롬프트 말미에 명시한다.

**3) 컨텍스트 최소화**
힌트 생성 시, 전체 대화 이력이 아닌 "현재 문제의 버그 코드 + 정답 코드 + 학생 현재 코드 + 힌트 레벨"만 전달한다. 불필요한 컨텍스트를 제거하여 토큰 사용량을 최소화하면서도 정확한 힌트를 생성한다.

**4) 모델 선택 최적화**
Gemini 2.0 Flash는 GPT-4o 대비 빠른 응답 속도와 낮은 비용을 제공하면서, 코드 생성/분석 품질은 교육용으로 충분하다. 대규모 추론이 필요 없는 교육용 프롬프트에 최적화된 선택이다.

**5) 결과 캐싱 가능 구조**
동일한 주제+난이도 조합에 대한 버그 챌린지를 DB에 저장하므로, 같은 주제를 다시 요청할 때 AI를 재호출하지 않고 기존 결과를 재사용할 수 있는 구조이다.
