## 1. 기획

### ■ 설정한 사용자는 누구이며, 해결하려는 구체적인 문제점/불편함은 무엇인가요?

솔직히 말하면, 저는 코딩을 배운 적이 없는 비전공자입니다. 개발 경험도, 부트캠프 수료 이력도 없습니다. 하지만 주변에서 국비지원 과정을 거친 사람들의 이야기를 들으면서, 코딩 교육의 문제는 오히려 코드를 모르는 사람 눈에 더 선명하게 보인다는 걸 느꼈습니다.

그래서 기술이 아니라 본질에 집중하기로 했습니다. "AI가 코드를 대신 짜주는 시대에, 개발자가 정말 잘해야 하는 건 뭘까?" 이 질문에서 출발했습니다.

BugHunter의 메인 사용자는 국비지원이나 부트캠프에서 코딩을 가르치는 강사이고, 서브 사용자는 그 수업을 듣는 수강생(비전공자 포함)입니다.

국비지원 교육 관련 커뮤니티(OKKY, DC인사이드, 브런치 등)의 후기들을 읽어보니, 수료생들이 실무에서 가장 힘들어하는 게 "디버깅"이었습니다. 코드를 짜는 건 어떻게든 하겠는데, 에러가 터지면 어디서부터 봐야 할지 모르겠다는 목소리가 많았습니다. 실제로 여러 연구에 따르면 현업 개발자들의 업무 시간 중 상당 부분이 기존 코드를 읽고 버그를 고치는 데 쓰인다고 합니다.

그런데 막상 교육 현장을 보면, 모든 커리큘럼이 "코드 짜는 법"에만 집중하고 있었습니다. 디버깅은 "알아서 하는 것"으로 취급받고, 체계적으로 훈련시키는 도구나 방법론이 없습니다. 특히 국비지원 과정은 강사 1명이 수강생 수십 명을 담당하는데, 디버깅처럼 사고 과정을 하나하나 봐줘야 하는 영역에서 개별 피드백이 물리적으로 불가능합니다.

여기에 AI 시대라는 변수가 겹칩니다. Copilot이나 Cursor 같은 도구가 코드 작성을 대신해줄수록, 개발자에게 남는 역할은 "AI가 만든 코드가 제대로 도는지 확인하고, 문제가 있으면 고치는 것"입니다. 그런데 지금 교육은 이 변화를 전혀 반영하지 못하고 있습니다. 저는 개발자가 아니기 때문에 코드 한 줄 짤 수 없지만, 이 간극만큼은 분명히 보였습니다. 그래서 이걸 메우고 싶었습니다.

### ■ 문제를 해결하기 위한 솔루션의 핵심 기능은 무엇인가요?

BugHunter는 "일부러 버그가 있는 코드를 AI가 만들어주고, 학생이 그걸 찾아서 고치는" 디버깅 훈련 플랫폼입니다. 접근 방식이 기존 교육 도구와 정반대인데, 보통은 "정답 코드를 짜라"고 하지만 여기서는 "틀린 코드를 고쳐라"고 합니다.

핵심 기능은 세 가지입니다.

첫째, AI가 버그 코드를 자동으로 만들어줍니다. 강사가 "오늘 주제는 반복문"이라고 입력만 하면, AI가 수 초 안에 초급(문법 오류), 중급(로직 오류), 고급(엣지케이스 미처리) 세 단계의 디버깅 문제를 생성합니다. 강사가 매번 직접 버그 코드를 고민해서 만들 필요가 없어집니다. 각 문제에는 딱 한 개의 버그만 들어가고, 테스트케이스도 같이 나오기 때문에 정답인지 바로 확인할 수 있습니다.

둘째, "답을 안 주는" 힌트 시스템입니다. 이게 ChatGPT랑 가장 다른 점입니다. ChatGPT에 코드를 붙여넣으면 바로 정답을 뱉어주는데, 그러면 학생은 아무것도 안 배웁니다. BugHunter의 AI는 절대 정답을 직접 알려주지 않습니다. 대신 3단계로 나눠서 사고를 유도합니다. 처음엔 "반복문 쪽을 한번 봐봐" 정도만 말해주고, 그래도 못 찾으면 "3번째 줄의 range 범위가 좀 이상하지 않아?"로 좁혀주고, 마지막에 "이건 off-by-one 에러야, range()의 끝값은 포함 안 된다는 거 알지?"까지 안내해줍니다. 교육학에서 말하는 "스캐폴딩"을 AI로 구현한 겁니다.

셋째, 강사용 대시보드와 학생 성장 리포트입니다. 강사가 수십 명을 직접 다 볼 수 없으니까, 대시보드에서 "우리 반은 반복문 버그는 잘 잡는데, 조건문 엣지케이스에서 많이 헤맨다" 같은 전체 현황을 한눈에 보여줍니다. 학생 입장에서는 자기가 얼마나 성장했는지 — 처음엔 문법 오류만 잡았는데, 이제 로직 오류도 잡는다 — 시각적으로 확인할 수 있어서 동기부여가 됩니다.

### ■ 이 솔루션이 도입되었을 때 기대되는 개선점이 무엇인가요?

가장 직접적인 개선은 강사의 시간입니다. 디버깅 실습 문제를 만드는 데 들이던 시간이 "주제 입력 10초"로 줄어듭니다. 수업 중에도 AI가 학생 수십 명에게 각각 힌트를 줄 수 있으니까, 강사는 AI 힌트로도 해결이 안 되는 소수의 학생에게만 집중하면 됩니다. 수십 명 전부 케어해야 하는 압박에서 벗어나는 겁니다.

수강생 측면에서는, "에러가 나면 ChatGPT에 던지고 끝"이던 패턴이 "직접 원인을 추적하고 고쳐보는" 경험으로 바뀝니다. CMU에서 비슷한 접근(HypoCompass)으로 초보자 대상 실험을 했을 때, 디버깅 테스트 점수가 약 12% 향상되었다는 결과가 있습니다. 직접 고민해서 버그를 잡아본 경험이 쌓이면, 실무에서도 에러 앞에서 당황하지 않게 됩니다.

결국 이 세 가지는 하나로 연결됩니다. 수강생이 스스로 버그를 잡을 수 있게 되면 강사의 부담이 줄고, 강사가 더 나은 수업에 집중할 수 있으면 교육기관 전체의 경쟁력이 올라갑니다. 수료생의 실무 적응력이 좋아지면 채용 기업의 피드백이 개선되고, "부트캠프 나왔는데 디버깅도 못 한다"는 말이 줄어드는 것만으로도 기관 평판에 큰 도움이 됩니다. 수강생을 위한 도구를 만들었지만, 결국 이건 강사를 위한 일이고, 교육기관을 위한 일이기도 합니다.

## 2. AI 활용 전략

### ■ 이 프로젝트에서 사용할 AI 도구(Claude Code, Cursor, Codex 등)와 모델은 무엇이며, 선택한 이유는 무엇인가요?

서비스 자체의 AI 엔진으로는 OpenAI GPT-4o를, 개발 과정에서는 Claude Code를 사용했습니다.

GPT-4o를 선택한 이유는 코드 이해력과 생성 품질이 교육용 프롬프트에 가장 적합했기 때문입니다. 특히 Python 코드의 버그를 의도적으로 만들어내는 것, 그리고 학생의 현재 코드를 분석해서 적절한 힌트를 주는 것 — 이 두 가지 핵심 기능에서 안정적인 결과를 보여줬습니다. JSON 형식 응답을 정확하게 지켜주는 점도 서비스 안정성에 중요했습니다.

Claude Code는 이 프로젝트 전체를 만드는 데 핵심적으로 활용했습니다. 솔직히 저는 코드를 직접 작성할 수 있는 사람이 아닙니다. 하지만 Claude Code와 대화하면서 제가 구상한 기능을 설명하고, 결과물을 확인하고, 방향을 조정하는 과정을 반복했습니다. Next.js 풀스택 구조 설계, Supabase 연동, API Route 작성, 그리고 GPT-4o에 전달할 프롬프트를 반복 테스트하며 최적화하는 과정까지 — "내가 코드를 짜는 게 아니라, AI와 함께 서비스를 설계하는 것"이라는 감각으로 진행했습니다. 비전공자인 제가 이 프로젝트를 완성할 수 있었던 건, 기술을 몰라도 문제를 정의하고 방향을 잡을 수만 있으면 AI가 구현을 도와준다는 걸 체감했기 때문입니다.

### ■ 각 AI 도구별 활용 전략을 작성해주세요.

**OpenAI GPT-4o — 서비스의 두뇌**

BugHunter에서 GPT-4o는 세 가지 서로 다른 역할을 합니다. 각 역할마다 전문화된 프롬프트를 설계해서, 하나의 모델이 마치 세 명의 전문가처럼 동작하게 만들었습니다.

첫 번째는 "버그 출제자" 역할입니다. 강사가 수업 주제를 입력하면, GPT-4o가 해당 주제에 맞는 버그 코드를 세 단계 난이도로 만듭니다. 여기서 가장 신경 쓴 건 프롬프트의 제약 조건입니다. "버그는 정확히 1개만", "난이도별 버그 유형 지정(초급=문법, 중급=로직, 고급=엣지케이스)", "테스트케이스 3개 이상 포함" 같은 규칙을 명시적으로 걸어둬서, AI가 학습 목적에 맞지 않는 문제를 만들지 않도록 했습니다. 응답은 엄격한 JSON 형식으로 받아서 파싱 에러를 방지했습니다.

두 번째는 "코칭 조교" 역할입니다. 학생이 힌트를 요청하면, 현재 문제의 버그 코드, 정답 코드, 학생이 지금 수정한 상태의 코드, 그리고 힌트 단계(1/2/3)를 GPT-4o에 전달합니다. 중요한 건 각 단계별로 "무엇을 말해도 되고, 무엇을 말하면 안 되는지"를 프롬프트에서 명확히 구분해둔 것입니다. 1단계에서는 줄 번호나 변수명을 절대 언급하지 말라고 하고, 3단계에서도 전체 정답 코드를 보여주지 말라고 강제합니다. 이 "말하지 않는 것"에 대한 제약이 ChatGPT 래핑과의 핵심 차이점입니다.

세 번째는 "채점관" 역할입니다. 학생이 코드를 수정해서 제출하면, 먼저 Pyodide(브라우저 내 Python 실행 엔진)로 테스트케이스를 돌려서 기계적으로 정답 여부를 확인하고, 그 다음 GPT-4o가 추가 피드백을 줍니다. 이렇게 이중으로 검증하는 이유는, 테스트는 통과했지만 코드 품질이 좋지 않은 경우(예: 하드코딩으로 테스트만 통과)도 잡아내기 위해서입니다.

전체 데이터 흐름을 정리하면 이렇습니다:

강사가 주제를 입력하면 → 버그 출제자가 문제를 만들고 → DB에 저장합니다. 학생이 문제를 풀다 막히면 → 코칭 조교가 단계별 힌트를 주고 → 학생이 수정해서 제출하면 → 채점관이 결과를 확인하고 피드백을 줍니다. 이 모든 결과가 DB에 쌓이면서 → 강사 대시보드와 학생 성장 리포트를 만듭니다.

**Claude Code — 개발 파트너**

Claude Code는 프로젝트를 처음 설계하는 단계부터 마지막 배포까지 전 과정에서 활용했습니다. 특히 유용했던 건 GPT-4o 프롬프트를 만들 때였습니다. "반복문 주제로 중급 버그 코드 만들어줘"라고 했을 때 GPT-4o가 뱉는 결과의 품질을 Claude Code 안에서 바로 확인하고, 프롬프트를 수정하고, 다시 테스트하는 사이클을 빠르게 돌릴 수 있었습니다.

저처럼 코드를 직접 작성하지 못하는 사람에게 Claude Code는 단순한 코드 생성 도구가 아니라, "내 머릿속 기획을 실제 서비스로 번역해주는 파트너"였습니다. 기능을 설명하면 구현해주고, 결과를 보고 "이건 좀 다르다"고 하면 수정하고, "이게 맞냐"고 물으면 근거를 들어 설명해주는 — 그런 협업이었습니다.

### ■ 토큰 낭비를 최소화하고 유지보수성 및 재현성을 높이기 위한 전략이 있다면 작성해주세요.

토큰 효율을 높이기 위해 몇 가지를 의식적으로 설계했습니다.

우선 프롬프트를 모듈화했습니다. 버그 생성용(`generate-bugs.ts`)과 힌트 생성용(`generate-hint.ts`)을 별도 파일로 분리해서, 프롬프트 수정이 필요할 때 해당 파일만 고치면 되게 만들었습니다. 각 프롬프트가 함수 형태라서, 같은 입력을 넣으면 같은 품질의 결과가 나옵니다.

응답 형식도 중요했습니다. AI에게 자유롭게 답하라고 하면 불필요한 설명이 붙어서 토큰이 낭비되고, 파싱도 어렵습니다. 그래서 버그 생성에서는 "JSON 배열만 응답해, 다른 텍스트 없이"를 강제했고, 힌트에서는 반대로 "마크다운이나 JSON 없이 자연스러운 문장만"으로 제한했습니다. 용도에 맞게 응답 형식을 통제하는 겁니다.

컨텍스트도 최소한으로 유지했습니다. 힌트를 생성할 때 이전 대화 이력을 전부 넘기는 대신, "지금 이 문제의 버그 코드 + 정답 코드 + 학생 현재 코드 + 힌트 단계" 네 가지만 전달합니다. 필요한 정보만 정확히 주는 게 토큰도 아끼고 힌트 품질도 올립니다.

마지막으로, 한 번 생성된 버그 챌린지는 DB에 저장되기 때문에, 같은 주제에 대해 반복 요청이 들어와도 AI를 다시 호출할 필요가 없습니다. MVP에서는 아직 캐싱 로직을 명시적으로 구현하진 않았지만, DB 구조 자체가 이미 재사용 가능한 형태로 설계되어 있습니다.
